One Interface, Four Backends

{{ app_name }} routes all configuration and asset IO through a single storage abstraction (sajha.core.storage). The same build runs unchanged on a laptop, on-prem, or in any cloud — you only change one config value.

local

Filesystem — laptop disk, EBS, or an EFS mount

s3

AWS S3 (and S3-compatible: MinIO, R2, Wasabi)

azure

Azure Blob Storage

gcs

Google Cloud Storage

Select a backend with storage.backend in config/application.yml (or the SAJHA_STORAGE_BACKEND environment variable). The default is local. Cloud SDKs are imported lazily, so the default install pulls none of them.

What Belongs on Object Storage

Object stores (S3, Azure, GCS) are ideal for read-mostly assets but are not a filesystem. Place data by class:

DataExamplesWhere
Read-mostly tool JSON configs, prompts, docs, reference CSVs, DuckDB source data Object store (s3/azure/gcs) — fine
Mutable state SQLite sajha.db, audit log, output cache, sessions EFS path or a managed service (RDS / DynamoDB) — never object storage

Object stores have no atomic append/rename, so running SQLite on them corrupts. Keep the database on a real filesystem (EFS) or switch db.type to postgresql (RDS).

local & EFS

The local backend reads and writes ordinary filesystem paths. It is the default and needs no cloud SDK. The path can be backed by a laptop disk, an EBS volume, or an Amazon EFS mount — to the application they are identical, because EFS is just a path, not a separate backend.

storage:
  backend: local
  base_dir: "."            # or an EFS mount, e.g. /mnt/efs/sajha

When to use EFS: a shared, POSIX, multi-instance filesystem on AWS — ideal for the mutable state that cannot live on S3 (the SQLite DB, audit log, cache) and for Studio-generated Python implementations that every instance must import.

Mount EFS (e.g. at /mnt/efs/sajha) and point base_dir (or the specific db.path / data dirs) at it. No code changes.

Amazon S3 (and S3-compatible)
storage:
  backend: s3
  s3:
    bucket: my-sajha-config
    prefix: sajha/
    region: us-east-1
    endpoint_url: ""        # set for MinIO / Cloudflare R2 / Wasabi
    cache_dir: /tmp/sajha-cache
    sync_interval: 60

Auth: the default boto3 credential chain. On EC2/ECS/EKS, attach an IAM role — no keys in config. Locally, standard AWS_* env vars or ~/.aws work.

S3-compatible: set endpoint_url to target MinIO, Cloudflare R2, or Wasabi with the same code.

Install: pip install boto3

Azure Blob Storage
storage:
  backend: azure
  azure:
    container: sajha-config
    account_url: https://<account>.blob.core.windows.net   # managed identity
    connection_string: ""   # alternative auth (keep in env, not here)
    prefix: sajha/
    cache_dir: /tmp/sajha-cache

Auth precedence: a connection string (AZURE_STORAGE_CONNECTION_STRING), otherwise account_url + DefaultAzureCredential (managed identity — no secrets in config).

Install: pip install azure-storage-blob azure-identity

Google Cloud Storage
storage:
  backend: gcs
  gcs:
    bucket: my-sajha-config
    project: my-gcp-project    # GOOGLE_CLOUD_PROJECT
    prefix: sajha/
    cache_dir: /tmp/sajha-cache

Auth: Application Default Credentials — the attached service account / workload identity on GCE/GKE/Cloud Run. No keys in config.

Install: pip install google-cloud-storage

How It Works
  • Local read-through cache. Cloud backends mirror fetched objects into cache_dir, so anything that needs a real file (e.g. importlib) still works.
  • Hot-reload. On local, a filesystem poller watches tool configs. On a cloud backend (no inotify), a sync manager polls the bucket every sync_interval seconds, mirrors changes to the cache, and reloads tools & prompts.
  • Tool implementations. Tool JSON configs live in the chosen backend; the Python implementation classes ship with the package and import locally. Studio-generated .py files are written locally for importlib — on multi-instance cloud deployments, place that directory on shared EFS so every instance can import them.
  • Migrating data. To move from local to a bucket, copy config/ and docs/ under the configured prefix (e.g. aws s3 sync ./config s3://my-sajha-config/sajha/config), then switch storage.backend.
